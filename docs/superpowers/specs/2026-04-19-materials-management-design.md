# 资料管理模块设计

- 日期：2026-04-19
- 状态：Draft for Review
- 适用范围：Socratiq 第一版资料管理模块

## 1. 背景

当前产品已经具备资料导入、异步处理、自动生成课程的基础能力，但用户看到的主要还是“导入流程”和“任务状态”。这会带来两个问题：

1. 资料、任务、课程三者的边界不清晰。
2. 用户缺少一个稳定的入口去查看和管理已上传资料。

本设计的目标不是一次性做出完整的内容后台，而是先建立一个清晰、可扩展的“资料视角”：

- `资料` 是独立资产。
- `任务` 是独立运行记录。
- `课程` 是由资料派生出来的学习产物。

同时保留当前最重要的默认流程：

- 用户导入一份资料。
- 系统自动创建资料处理任务。
- 资料处理成功后，系统自动创建课程生成任务。

## 2. 设计目标

第一版资料管理模块需要解决以下问题：

- 给用户一个明确的 `资料` 入口，而不是“导入历史”或纯任务列表。
- 让用户能快速找到自己上传过的资料，并理解每份资料当前处于什么状态。
- 让用户看清资料处理或课程生成卡在哪一步，但不把页面做成复杂任务后台。
- 让已完成资料有明确的主动作：`进入课程`。
- 在产品和数据模型上为 `资料 / 任务 / 课程 解耦` 打基础。

## 3. 非目标

以下内容明确不属于第一版：

- 多资料手动组课
- 用户手动发起课程生成
- 标签、分组、收藏、归档
- 批量操作
- 独立的完整任务中心页面
- 原始日志查看器
- 资料版本历史
- 课程编辑型管理视图

## 4. 核心模型

### 4.1 资料

资料是用户上传或录入的原始学习资产，是本模块中的主对象。

资料具备以下特征：

- 可独立存在，不依赖课程才“有意义”。
- 可以被复用。
- 可以在未来参与单资料或多资料组课。
- 需要有自己稳定的展示状态，而不是直接等同于某个任务状态。

### 4.2 任务

任务是系统对资料或课程执行的异步运行记录，用来描述过程，而不是充当资产本身。

第一版至少要支持两类任务概念：

- `资料处理任务`
- `课程生成任务`

任务应该能表达：

- 任务类型
- 归属对象
- 当前状态
- 当前阶段
- 失败摘要
- 起止时间
- 扩展元数据

### 4.3 课程

课程是由一份或多份资料派生出来的学习产品。

第一版虽然不做多资料手动组课，但模型必须允许未来扩展到：

- 多份资料可以组合生成一门课程。
- 一份资料可以参与多门课程的生成。

因此课程不应再被视为资料的唯一结果字段，而应是资料的下游关联产物。

## 5. 信息架构

### 5.1 导航

建议将当前侧边栏中的 `导入资料` 调整为 `资料`。

原因：

- 用户需要的是长期存在的资料入口，而不是单次导入动作入口。
- “导入资料”更适合作为页面内主按钮，不适合作为一级导航主概念。

导航调整后：

- 一级导航：`首页` / `资料` / `设置`
- `导入资料` 作为资料页右上角主按钮
- 现有 `/import` 页面保留，作为新建资料入口
- 第一版继续沿用 `/sources` 作为资料页路由承载，页面文案统一改成 `资料`

### 5.2 模块定位

第一版采用“方案 2”：

- 主模块是 `资料中心`
- 任务作为资料的二级运行信息存在
- 不单独做完整任务中心

这可以同时满足两个要求：

- 用户先有一个稳定入口查看资料
- 后续仍能平滑演进为资料中心 + 任务中心的完整体系

## 6. 页面结构

第一版资料页采用三层结构：

1. 顶部操作区
2. 资料列表区
3. 资料详情抽屉

### 6.1 顶部操作区

顶部区域包含：

- 页面标题：`资料`
- 页面说明：管理你导入过的学习资料，并查看它们的处理状态和课程产出
- 主按钮：`导入资料`

第一版筛选控件建议包含：

- 搜索：按标题 / URL / 文件名
- 状态筛选：`全部 / 处理中 / 已完成 / 失败`
- 类型筛选：`全部 / B站 / YouTube / PDF`
- 排序：
  - 默认：`可行动性优先`
  - 可切换：`最近导入`

### 6.2 资料列表区

页面主体是资料列表，而不是双 Tab。

原因：

- 第一版的主任务是“让用户看资料”，不是“做任务后台”。
- 如果把任务抬成平级主视图，会让任务重新成为主角。

默认排序策略采用 `可行动性优先`：

1. 失败
2. 处理中
3. 已完成
4. 每组内部按最近更新时间倒序

这样可以让最需要处理和关注的资料优先暴露。

### 6.3 资料详情抽屉

点击资料项后，从右侧打开详情抽屉，不跳转新页面。

这样做的理由：

- 保持“浏览列表 -> 查看某份资料 -> 返回列表”的顺畅节奏。
- 避免第一版为了详情信息引入新的路由复杂度。
- 便于未来在列表页上持续迭代筛选和多态展示。

## 7. 资料卡片设计

每条资料卡片包含以下信息：

- 标题
  - 优先显示资料标题
  - 缺失时回退到 URL 或原文件名
- 来源信息
  - 来源类型图标
  - 来源类型文案
  - 导入时间
- 当前运行状态
  - 状态徽标
  - 当前阶段文案
  - 失败摘要或复用信息
- 主动作
  - 已完成：`进入课程`
  - 处理中：无强动作，仅展示阶段信息
  - 失败：`查看原因`
- 次级动作
  - 删除
  - 重试或重新处理（若后端提供）
  - 查看详情

卡片不直接展示大量课程信息，只保留必要的结果关联，避免资料卡片被做成课程卡片。

## 8. 详情抽屉设计

详情抽屉分为四个区块：

### 8.1 基础信息

- 标题
- 来源类型
- 原始链接或文件名
- 导入时间

### 8.2 当前状态

- 当前资料状态
- 当前阶段
- 最近更新时间
- 是否复用已有结果

### 8.3 运行记录

第一版不做完整日志流，只展示最近一次关键运行信息：

- 最近资料处理任务的状态和阶段
- 最近课程生成任务的状态和阶段
- 最近一次失败原因

### 8.4 关联结果

- 若已有课程：展示“已生成课程”并提供 `进入课程`
- 若暂无课程：展示“尚未生成课程”

详情抽屉承担“解释状态”的职责，而不是变成一个复杂的运维面板。

## 9. 状态模型

资料、任务、课程必须各自拥有真实状态，前端负责做面向用户的聚合展示。

### 9.1 用户可见资料状态

第一版建议采用以下资料层展示状态：

- `处理中`
  - 表示存在未完成任务
- `资料可用`
  - 表示资料处理已完成
- `课程生成中`
  - 表示资料已可用，但课程生成任务仍在进行
- `课程生成失败`
  - 表示资料可用，但课程产出失败
- `资料处理失败`
  - 表示资料本身尚不可用

### 9.2 状态聚合原则

- 资料处理失败不等于整条链路彻底无效，资料仍应保留并可见。
- 课程生成失败不应把资料打回“不可用”。
- 课程是否已生成应作为结果态存在，而不是覆盖资料主状态。

## 10. 交互流程

第一版主流程如下：

1. 用户导入资料。
2. 系统创建资料记录。
3. 系统自动创建资料处理任务。
4. 资料处理任务成功后，系统自动创建课程生成任务。
5. 资料页根据资料与任务的聚合状态展示当前情况。
6. 若课程生成成功，资料卡片提供 `进入课程`。

这一流程延续了当前默认自动化体验，但在概念上完成解耦：

- 自动生成课程是默认编排，不是资料存在的前提。
- 任务是过程记录，不是资料本身。

## 11. 后端与数据模型调整

第一版建议至少补齐以下能力。

### 11.1 增加独立任务记录

不再只依赖 Celery 状态和 `source.status` 组合表达全流程。

建议引入正式 `Task` 记录，至少包含：

- `type`
- `subject_type`
- `subject_id`
- `status`
- `stage`
- `error_summary`
- `started_at`
- `finished_at`
- `metadata`

### 11.2 资料接口返回 summary 结构

资料列表接口和详情接口应返回轻量聚合信息，例如：

- `latest_task_summary`
- `latest_processing_task_summary`
- `latest_course_task_summary`
- `course_count`
- `latest_course_id`

这样前端可以以“资料视角”渲染，而不需要自己拼凑底层运行状态。

### 11.3 去除资料与课程的硬耦合字段

不应继续依赖 `source.metadata_.course_id` 之类的耦合字段来表达正式关系。

课程与资料的关联应通过正式关系表和任务结果表示，保证未来可以扩展到多资料组课。

## 12. 异常处理

异常处理分三层。

### 12.1 列表层

只显示短错误摘要，例如：

- `课程生成失败`
- `字幕提取失败`

### 12.2 详情层

展示更明确的信息：

- 失败发生在哪个任务阶段
- 最近一次失败原因
- 是否可重试

### 12.3 系统层

保留更完整的底层任务信息和日志能力，但第一版不直接向用户暴露原始日志。

## 13. 前端实现边界

第一版明确包含：

- 正式的 `资料` 入口
- 资料列表页
- 搜索 / 类型筛选 / 状态筛选 / 排序
- 资料卡片状态展示
- 资料详情抽屉
- 已完成资料的 `进入课程`
- 失败原因查看
- 处理阶段可视化展示

第一版明确不包含：

- 多资料手动组课
- 手动创建课程生成任务
- 标签、分组、收藏、归档
- 批量操作
- 独立完整任务中心
- 原始日志查看器
- 资料版本历史

## 14. 测试策略

第一版至少需要覆盖以下测试面。

### 14.1 后端接口测试

- 资料列表接口返回新的 summary 字段
- 资料详情接口返回任务摘要和课程摘要

### 14.2 前端状态映射测试

- 资料状态、任务状态、课程状态如何映射成用户可见状态
- 列表排序是否符合“可行动性优先”

### 14.3 页面交互测试

- 搜索和筛选
- 打开详情抽屉
- `进入课程`
- 错误摘要与详情展示

### 14.4 回归测试

- 保持原有 `/import` 导入流程可用
- 自动资料处理与自动课程生成流程不被破坏
- 现有首页和课程入口不回退

## 15. 推荐实现顺序

建议按以下顺序推进，而不是先大规模重构底层：

1. 调整产品入口
   - 侧边栏由 `导入资料` 转为 `资料`
   - 资料页右上角保留 `导入资料`
2. 搭出资料页骨架
   - 列表、筛选、状态卡片、详情抽屉
3. 补齐后端 summary 结构
   - 先满足资料视角渲染
4. 梳理资料、任务、课程的状态聚合规则
   - 避免继续把课程状态硬塞进资料主状态
5. 后续阶段再演进独立任务中心

这样可以先尽快提供“资料入口”的产品价值，同时保留解耦方向。

## 16. 最终建议

第一版应把当前的“导入历史”升级为真正的 `资料` 模块，而不是继续围绕“导入日志”组织界面。

这个模块的成功标准不是功能有多全，而是用户能自然理解：

- 我上传的东西叫 `资料`
- 系统在后台跑的是 `任务`
- 最终产出的是 `课程`

这三个概念一旦清楚，后续无论做任务中心、资料复用、还是多资料组课，产品都会更稳定。
